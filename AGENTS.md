# AGENTS.md - react-rasenmaeher-ui-v2

## Purpose
This repo is the web frontend shell for Deploy App (RASENMAEHER). It owns the
enrollment flow, admin tools, shared layout, theming, localization, browser
guardrails, and the container UI that loads product integrations.

It does **not** own product backend logic, Docker wiring, volume plumbing, or
most product-specific UI code. Those come in through `rmapi`, nginx, and the
product repos.

## Stack & Key Technologies
- **Language:** TypeScript (strict mode)
- **Framework:** React 18.3 + Vite 5
- **Routing:** `@tanstack/react-router` with generated `src/routeTree.gen.ts`
- **Data fetching:** `react-query` v3
- **UI layer:** Radix UI primitives + local wrappers under `src/components/ui`
- **Styling:** Tailwind CSS 4 via `@tailwindcss/vite`
- **Localization:** `i18next` + base locale files + theme/runtime overrides
- **Package manager:** `pnpm` only
- **Micro-frontends:** Module Federation via `@module-federation/vite`
- **Delivery model:** Static shell served by nginx, product assets/components exposed under `/ui/<shortname>/...`

## Development Setup

**The UI cannot be run standalone in a meaningful way.** The full experience requires the
`docker-rasenmaeher-integration` stack. Under `rmdev`, the UI is served at
`https://localmaeher.dev.pvarki.fi:4439` with hot-reload enabled.

```bash
# Primary: run via the integration stack (from docker-rasenmaeher-integration/)
rmdev build
rmdev up
# UI available at: https://localmaeher.dev.pvarki.fi:4439
```

Standalone Vite (`pnpm run dev`) is only useful for isolated shell, route, theme, or
component work where you do not need mTLS, the proxy layer, product assets, or real auth.
It will not render product cards or run enrollment flows correctly.

```bash
# Standalone shell dev only — limited use
pnpm install
pnpm run dev
```

Build-time env currently used in this repo:
- `VITE_THEME`
- `SERVER_DOMAIN`

## Workspace Context

git submodule. UI code changes belong here; compose, nginx, and shared volume changes
belong in the orchestration repo. Keep those commits separate.

## Architecture Notes
**Bootstrap path:** `src/main.tsx` loads theme data, applies CSS variables, installs
runtime localization overrides, creates the `react-query` client, mounts the browser
guard, and wraps the app in `UserTypeFetcher` before rendering routes.

**Routing:** Routes are file-based under `src/routes/`. `src/routeTree.gen.ts` is
generated by the TanStack Router plugin. Do not hand-edit generated route metadata.

**Auth model:** The shell determines auth state through `rmapi`, not by talking directly
to product services. `UserTypeFetcher` calls:
- `/api/v1/check-auth/mtls_or_jwt`
- `/api/v1/check-auth/validuser`
- `/api/v1/check-auth/validuser/admin`

It uses JWT from local storage when available and also relies on proxied cookies.
Preserve the current allowlisted unauthenticated routes unless the auth flow is being
changed intentionally.

**mTLS redirect behavior:** Valid users landing on `/` on the non-mTLS host may be
redirected to `mtls.<host>`. This behavior is intentional and lives in the root route
layout. Be careful when changing navigation or host-based logic.

**Theme system:** Theme assets do **not** live in `src/assets/`. They live under
`public/themes/<theme>/` and are loaded through Vite plugins plus runtime helpers.
Each theme may provide:
- `theme.json`
- `manifest.json`
- `assets/`
- `icons/`
- `localization/en.json`, `fi.json`, `sv.json`

Use the theme loaders and `virtual-theme-config`; do not hardcode theme file paths.

**Localization model:** Base UI strings live in `src/locales/`. Theme-specific runtime
overrides can replace strings per language. If you add user-facing text, keep the base
locale files aligned and make sure theme overrides still fall back cleanly.

**Product loading contract:** The shell is intentionally generic. It learns what products
exist from `rmapi` and loads product content dynamically.

User-facing product data currently comes from:
- `/api/v2/descriptions/<language>`
- `/api/v2/instructions/data/<product>`

Admin-facing product data currently comes from:
- `/api/v2/admin/descriptions/<language>`

Each product description includes a `component` object with:
- `type`: `link`, `markdown`, or `component`
- `ref`: the URL or asset path to load

The main UI should stay mostly product-agnostic. Adding a new product should usually be a
backend/integration concern first, not a shell rewrite here.

**Federated product UI:** For `component` products, the shell loads Module Federation
remotes from `/ui/<shortname>/remoteEntry.js`, registers the remote as
`<shortname>-integration`, and imports `remote-ui`. If that load fails, the shell falls
back to a local fallback component.

**Proxy and volume boundary:** The frontend must not assume it can call product services
directly. In the real system, `rmapi` sits in the middle, nginx exposes the browser-facing
paths, and shared volumes make product UI assets available. If a product needs markdown,
JSON data, or a federated bundle, the expected path is through `rmapi` and `/ui/<shortname>/...`,
not through a direct browser call to the Python service container.

**Serving product assets:** Product-owned static assets are expected under
`/ui/<shortname>/...`. That path exists because the integration/orchestration side copies
files into the shared `ui_files` volume and exposes them through nginx. This repo consumes
that contract; it does not create it.

## Modular UI Integration Spec

This section covers what product integrations must implement to appear in the shell.
Full sequence diagrams and Dockerfile examples are in
`rasenmaeher-ui-v2-wiki/Home.md`.

### Component Types

| Type | `component.ref` value | How it renders |
|---|---|---|
| `link` | Any URL | Opens external link; no shell embedding |
| `markdown` (nginx) | `/ui/{shortname}/{file.md}` | Fetched from `ui_files` volume via nginx |
| `markdown` (proxy) | Proxy path via `rmapi` | Shell calls `/api/v1/product/proxy/{shortname}/{route}` |
| `component` | `/ui/{shortname}/remoteEntry.js` | Module Federation remote loaded by the shell |

### Required Integration API Routes

Every integration needs these endpoints under its own `/api/v2` namespace:

**`GET /api/v2/description/{language}`** — Required for the product card to appear:
```python
class ProductDescriptionExtended(BaseModel):
    shortname: str        # slug used in URLs
    title: str
    icon: Optional[str]   # URL to icon
    description: str
    language: str
    docs: Optional[str]   # link to documentation
    component: ProductComponent  # type + ref

class ProductComponent(BaseModel):
    type: Literal["link", "markdown", "component"]
    ref: str
```

**`POST /api/v2/clients/data/`** — Optional; provides dynamic data passed to component as props:
```python
class ClientInstructionResponse(BaseModel):
    data: ClientInstructionData  # arbitrary dict
```

Admin equivalents: `GET /api/v2/admin/description/{language}` and
`POST /api/v2/admin/clients/data/{product}`.
**Admin status:** Only `link` type is tested for admin cards. `component` and `markdown`
admin types have not been validated end-to-end.

### Serving Files (Docker Setup)

Product UI assets land in the shared `ui_files` Docker volume and become reachable at
`/ui/{shortname}/...` via nginx. Minimum changes per integration:

**`docker-compose.yml`** — add `ui_files:/ui_files` under `volumes` for the integration service.

**`Dockerfile`** — add to the relevant build stages:

```dockerfile
# builder_base stage: install Node + pnpm
RUN curl -fsSL https://deb.nodesource.com/setup_20.x | bash - \
    && apt-get install -y nodejs \
    && corepack enable \
    && corepack prepare pnpm@latest --activate

# production_build stage: build UI
COPY ./ui /ui/
WORKDIR /ui
RUN CI=true pnpm install && pnpm build
RUN mkdir -p /ui_build && cp -r dist/* /ui_build/
WORKDIR /app

# production stage: carry build output
COPY --from=production_build /ui_build /ui_build
```

**`docker/entrypoint.sh`** — copy assets into the shared volume at container startup:
```bash
mkdir -p /ui_files/{shortname}
if [ -d "/ui_build" ]; then
    cp -r /ui_build/* /ui_files/{shortname}/
fi
```

### Dev Hot-Reload Entrypoint

For `rmdev`, use `pnpm build --watch` piped to the volume so browser refreshes pick up changes:

```bash
# docker/entrypoint-dev.sh
pnpm --dir /ui install
mkdir -p /ui_files/{shortname}
pnpm --dir /ui build --outDir /ui_files/{shortname} --watch &
# ... then start the Python service as usual
```

### Asset Path Rule

Inside federated components, **always** prefix asset references with `/ui/{shortname}/`:
```
# Correct
/ui/tak/taistelija.png

# Wrong — breaks in production because assets are served from the volume path
./taistelija.png
```

Place assets in the component project's `public/` folder; the build copies them into `dist/`.

### Getting Data into a Component

The shell automatically calls `POST /api/v2/clients/data/` and passes the response as props
to the federated component. Components can also make direct API calls through `rmapi`
(the browser cannot reach product services directly — use the proxy at
`/api/v1/product/proxy/{shortname}/{route}`).

For local dev without the full stack, paste the response from
`https://localmaeher.dev.pvarki.fi:4439/api/v2/instructions/data/{shortname}`
into `src/main.tsx` as `SAMPLE_DATA`.

### Federated Component Vite Config (product side)

The remote project's `vite.config.ts` must expose `remote-ui` as the federation entry:

```typescript
import { federation } from "@module-federation/vite";

export default defineConfig({
  plugins: [
    react(),
    federation({
      name: "myproduct-integration",  // unique name for the remote
      filename: "remoteEntry.js",
      exposes: {
        "./remote-ui": "./src/App",   // must be "remote-ui" — the shell looks for this key
      },
      shared: ["react", "react-dom"],
    }),
  ],
  base: "/ui/myproduct/",  // must match the volume path
  build: { target: "esnext" },
});
```

The existing product integrations that have a `ui/` folder are good copy-paste starting
points — explore the submodule directories to find them.

## Change Routing
If the change is about shell layout, routes, shared components, theme loading, browser
guard behavior, or how the main UI hosts products, edit this repo.

If the change is about enrollment logic, auth semantics, product description aggregation,
instruction payloads, or proxy behavior, edit `python-rasenmaeher-api` / `rmapi`.

If the change is about Docker Compose, nginx exposure, `/ui_files` volume wiring, domains,
or how product UI files become reachable in the browser, edit
`docker-rasenmaeher-integration`.

If the change is about the actual product-specific React component, markdown, or assets for
one product, edit that product repo's `ui/` folder instead of teaching the shell about that
product directly.

## Debugging Map
- Product card missing: check the `rmapi` descriptions endpoint before changing card rendering.
- Product modal opens fallback: verify `/ui/<shortname>/remoteEntry.js` exists and the remote exposes `remote-ui`.
- Markdown product is blank: verify `component.ref` and whether the file is really reachable through proxy or `/ui/<shortname>/...`.
- Product component has no data: check `/api/v2/instructions/data/<product>` and auth requirements.
- Works in standalone Vite but not in compose: inspect nginx routes, `ui_files` volume copy logic, and integration build output.
- Auth redirects loop: inspect `UserTypeFetcher`, root route auth exemptions, and host-based mTLS redirect logic.

## Common Agent Pitfalls
1. **Use `pnpm`, not `npm` or `yarn`.** The lockfile is `pnpm-lock.yaml`. Do not create `package-lock.json`.
2. **Do not edit `src/routeTree.gen.ts` manually.** It is generated.
3. **Do not hardcode `localhost`, service container names, or product API URLs in browser fetches.** Use the existing relative `/api/...` and `/ui/...` contracts.
4. **Do not make the shell product-aware unless there is a real shell concern.** Most new products should appear through metadata and remote assets, not new hardcoded UI branches.
5. **Do not bypass `rmapi` for integration API calls.** Browser requests do not carry the internal network topology, proxy behavior, or certificates the same way the backend does.
6. **Do not treat Docker volumes as a frontend concern.** The browser only sees proxied paths such as `/ui/<shortname>/...`; volume copying belongs to the integration side.
7. **Do not remove module federation because a product currently works as markdown or link.** The shell must keep supporting `component`, `markdown`, and `link` modes.
8. **Do not document or run a nonexistent `pnpm run type-check`.** Use `pnpm run build` and `pnpm run lint` instead.
9. **Do not mix public and admin product endpoints.** User flows and admin flows already have separate API routes.
10. **Do not assume standalone Vite proves production correctness.** Product remotes, nginx, mTLS, and shared assets are best validated through the orchestration repo.
11. **Do not move theme assets into source code paths.** Theme data belongs under `public/themes/<theme>/`.
12. **React Compiler is intentionally disabled.** Do not enable it without team discussion.
